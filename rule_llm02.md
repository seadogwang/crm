# 规则引擎补充设计文档：AI 对话式规则配置 V3（需求澄清与动态表单）
> **关联主文档**：基于AI的下一代全渠道忠诚度管理SaaS平台 v7.3\
> **版本**：3.0\
> **设计目标**：在 V2（对话+表单混合）基础上，增加**需求澄清阶段**，确保 AI 在生成表单前完整理解用户业务意图，尤其针对“指定商品多倍积分”等高频但语义含糊的场景。
***
## 一、设计背景与核心问题
### 1.1 设计背景
在 V2 设计中，AI 在理解用户初始需求后，如果信息不完整，会直接生成一个包含所有必填字段的**澄清表单**。但在实际业务场景中，特别是“指定商品多倍积分”这类需求，存在以下问题：
| 问题           | 具体表现                    | 影响               |
| ------------ | ----------------------- | ---------------- |
| **语义歧义**     | “指定商品多倍积分”至少有5种不同的业务解释  | AI直接生成表单可能遗漏关键逻辑 |
| **用户不知道填什么** | 表单字段如“非指定商品处理方式”用户可能不理解 | 填写质量差            |
| **业务逻辑不明确**  | 是否叠加、是否替代普通积分等问题未澄清     | 生成的规则可能不符合业务预期   |
| **前后端数据依赖**  | 商品列表、品类列表等字典数据需要从后端获取   | 纯对话无法完成          |
### 1.2 核心设计原则
| 原则          | 说明                                                      |
| ----------- | ------------------------------------------------------- |
| **澄清与填写分离** | 先对话澄清业务逻辑（为什么做），再表单填写具体数值（做什么）                          |
| **业务语言优先**  | 澄清问题使用业务语言（“非指定商品怎么处理积分？”），而非技术语言（“请输入 defaultRule 字段”） |
| **动态生成表单**  | 表单根据澄清结果动态生成，不同场景显示不同字段                                 |
| **字典数据控件化** | 商品、品类等数据通过 API 获取并渲染为选择器                                |

## 三、关键业务逻辑澄清
### 3.1 “指定商品多倍积分”需求必须澄清的6个核心问题
| 序号 | 澄清问题（业务表达）       | 澄清目的           | 选项/示例               |
| -- | ---------------- | -------------- | ------------------- |
| 1  | **非指定商品怎么处理积分？** | 确定普通商品的积分策略    | ①正常累积（1倍）②不累积 ③固定积分 |
| 2  | **活动时间范围是什么时候？** | 确定有效时间段        | 开始时间 → 结束时间         |
| 3  | **积分计算以哪个时间为准？** | 明确判断订单在活动期内的基准 | 下单时间 / 付款时间 / 签收时间  |
| 4  | **多倍积分是多少倍？**    | 确定具体倍数         | 2倍 / 3倍 / 自定义       |
| 5  | **是否需要最低消费门槛？**  | 避免小额订单滥用       | 满X元 / 不限制           |
| 6  | **是否限制会员等级？**    | 精准运营           | 全部会员 / 黄金及以上 / 仅铂金  |
### 3.2 澄清对话示例
```text
用户：618活动，指定商品多倍积分
AI：收到需求。我想确认几个关键点：
1. 非指定商品（即参与活动的商品之外的普通商品）的积分怎么处理？
   A. 正常累积（1倍）
   B. 不累积积分
   C. 固定积分（如每件10分）
2. 本次活动的时间范围是？
3. 多倍积分是多少倍？（如2倍、3倍）
用户：非指定商品正常累积1倍。时间6月20日到30日。指定商品2倍积分。
AI：明白了。再确认一下：
1. 积分计算以下单时间、付款时间还是签收时间为准？
2. 是否需要最低消费门槛（如：实付金额满100元）？
3. 是否只针对特定等级的会员？
用户：付款时间。不需要门槛。全部会员。
AI：信息已完整。我将为您生成规则配置表单，请填写具体数值。
```

## 五、与 V2 设计的主要差异
| 对比项         | V2（原有设计）      | V3（本补充设计）             |
| ----------- | ------------- | --------------------- |
| **问题收集方式**  | 一步生成表单，用户直接填写 | **先对话澄清业务逻辑，再生成动态表单** |
| **表单生成时机**  | 用户输入后立即生成     | **澄清完成后生成**           |
| **表单字段来源**  | 固定的通用模板       | **根据澄清结果动态定制**        |
| **非指定商品处理** | 未明确           | **必须在澄清阶段明确**         |
| **时间语义**    | 表单中提供选项       | **在澄清中明确后，表单直接展示选项**  |
| **字典数据控件**  | 仅文本提示         | **API 驱动的选择器**        |
| **交互效率**    | 用户可能不知道填什么    | **用户先理解业务逻辑再填具体值**    |

## 六、API 接口补充
### 6.1 澄清对话接口（流式）
```text
POST /api/rules/ai/clarify
请求：
{
  "sessionId": "sess_xxx",
  "message": "618活动，指定商品多倍积分"
}
响应（SSE流式）：
event: text
data: {"type": "text", "content": "收到需求。我想确认几个关键点："}
event: text
data: {"type": "text", "content": "\n1. 非指定商品..."}
event: question
data: {
  "type": "question",
  "id": "q1",
  "question": "非指定商品（即参与活动的商品之外的普通商品）的积分怎么处理？",
  "options": [
    { "value": "NORMAL_1X", "label": "正常累积（1倍）" },
    { "value": "NO_REWARD", "label": "不累积积分" },
    { "value": "FIXED_POINTS", "label": "固定积分（每件10分）" }
  ],
  "required": true
}
event: done
data: {"type": "done"}
```
### 6.2 提交澄清结果
```text
POST /api/rules/ai/clarify/submit
请求：
{
  "sessionId": "sess_xxx",
  "answers": {
    "q1": "NORMAL_1X",
    "q2": "2026-06-20 00:00 至 2026-06-30 23:59",
    "q3": "2"
  }
}
响应：
{
  "status": "CLARIFIED",
  "formSchema": { ... }   // 生成的动态表单
}
```
### 6.3 提交表单（同V2）
```text
POST /api/rules/ai/submit-form
```

## 七、开发实施要点
| 序号 | 任务           | 说明                                                |
| -- | ------------ | ------------------------------------------------- |
| 1  | **后端对话状态管理** | 扩展会话状态，区分 `COLLECTING`（澄清中）和 `CLARIFIED`（已澄清，待填表） |
| 2  | **澄清问题模板**   | 针对“指定商品多倍积分”等高频场景，预置澄清问题模板，避免AI自由发挥导致遗漏           |
| 3  | **动态表单生成器**  | 根据澄清结果动态生成 `formSchema`，而非固定模板                    |
| 4  | **字典数据 API** | 新增商品搜索、品类树、积分类型、等级列表、渠道列表等接口                      |
| 5  | **流式输出增强**   | 支持 `question` 类型事件，前端可渲染选项按钮而非纯文本                 |
| 6  | **前端表单渲染器**  | 支持所有字段类型（特别是 `sku_selector`、`category_selector`）  |

## 八、总结
本补充设计在 V2（对话+表单混合）基础上，增加了**需求澄清阶段**：
1. **先对话澄清业务逻辑**：通过 2-3 轮对话，明确核心业务决策（非指定商品处理、时间基准、会员等级限制等）
2. **再生成动态表单**：基于澄清结果，生成包含具体数值字段的结构化表单
3. **最后表单填写**：用户只需选择/填写具体数值，无需理解复杂业务逻辑
关键改进：**表单不是固定的，而是根据澄清结果动态生成的**，确保用户看到的每个字段都是必要的、可理解的。这有效解决了“用户不知道填什么”的问题。

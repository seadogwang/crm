## 缺失 1：A/B 测试与实验能力（Optimization 层）
### 问题
目前系统只能“执行一个策略”，无法在线上做 **A/B 测试对比**（如：邮件A vs 邮件B，哪个转化率更高）。这是营销平台最基本的能力之一。
### 应该放在哪里？
* 第4章（Simulation & Optimization）扩展
* 或者第5章（Execution）的节点级控制
### 需要定义的内容
1. **实验配置**：在画布中支持“实验节点”（EXPERIMENT），定义多个变体（Variant A/B/C）。
2. **流量分配**：按比例（如 50%/50%）将用户分配到不同分支。
3. **目标指标**：实验的目标是优化什么？（点击率、转化率、收入）
4. **运行时长**：实验运行多久？何时出结果？
5. **自动选择胜者**：实验结束后，自动将胜者推送给剩余用户（或自动切换默认分支）。
6. **统计显著性计算**：使用 T 检验或贝叶斯方法判断结果可信度。
**例子**：
```text
[人群筛选] → [EXPERIMENT: 主题行A vs 主题行B]
                   ├── 50% → [发送邮件(主题A)]
                   └── 50% → [发送邮件(主题B)]
```

## 缺失 2：用户偏好与退订管理（Consent & Preference）
### 问题
目前系统**没有任何关于“用户同意/退订”的设计**。营销平台必须支持：
* 用户退订所有营销邮件（全局退订）
* 用户选择接收渠道（只收邮件，不收短信）
* 用户选择接收品类（只收服装促销，不收3C促销）
* GDPR/CCPA 合规的数据删除请求
### 应该放在哪里？
* 第13章（Content & Compliance Governance）扩展
* 或者作为独立的“用户偏好中心”
### 需要定义的内容
1. **偏好数据模型**：
   sql
   复制
   下载
   ```
   CREATE TABLE campaign_user_preference (
       member_id VARCHAR(64) PRIMARY KEY,
       program_code VARCHAR(32) NOT NULL,
       -- 全局退订
       global_unsubscribe BOOLEAN DEFAULT FALSE,
       unsubscribe_reason VARCHAR(64),
       unsubscribe_at TIMESTAMPTZ,
       -- 渠道偏好
       email_opt_in BOOLEAN DEFAULT TRUE,
       sms_opt_in BOOLEAN DEFAULT TRUE,
       push_opt_in BOOLEAN DEFAULT TRUE,
       -- 品类偏好（JSON 灵活扩展）
       category_preferences JSONB,  -- {"include": ["服装", "美妆"], "exclude": ["3C"]}
       -- 时间偏好
       preferred_time_start TIME,
       preferred_time_end TIME,
       timezone VARCHAR(64),
       updated_at TIMESTAMPTZ
   );
   ```
2. **Worker 侧强制检查**：任何发送动作前，必须检查 `global_unsubscribe` 和渠道 `opt_in`。
3. **退订处理流程**：从 Campaign 点击退订链接 → 更新偏好 → 停止后续营销。
4. **数据删除请求（GDPR）**：接受删除请求 → 匿名化或删除用户所有数据。
***
## 缺失 3：预算节奏控制（Budget Pacing）
### 问题
目前决策引擎只做一次性预算分配，但没有**运行时预算控制**（即：营销预算如何在活动期间“匀速”消耗）。
业务场景：
* 总预算 10 万元，活动持续 30 天，希望每天花 3,333 元，避免第一天就花光。
* 如果某天转化效果特别好，允许自动加速消耗。
### 应该放在哪里？
* 第3章（Decision Engine）扩展
### 需要定义的内容
1. **匀速消耗（Daily Pacing）**：
   * 每日预算上限 = 总预算 / 活动天数
   * Worker 发送前检查“今日已花费 < 今日上限”
2. **动态调速（Dynamic Pacing）**：
   * 根据实时 CTR/转化率，自动调整次日预算（效果好的 Campaign 给更多预算）
3. **预算警报**：
   * 当日消耗达到 80% → 警告
   * 当日消耗达到 100% → 暂停今日剩余任务
***
## 缺失 4：活动日历与冲突检测（Campaign Calendar）
### 问题
目前没有全局视图管理“所有活动的时间安排”。多个 Campaign 可能在同一时间窗口同时运行，导致：
* 同一用户被频繁轰炸
* 渠道容量瞬间超载
### 应该放在哪里？
* 第1章（Planning Workspace）扩展
* 或第3章（Decision Engine）扩展
### 需要定义的内容
1. **日历视图**：按时间展示所有 Campaign 的执行计划。
2. **冲突高亮**：相同用户群 + 相同渠道 + 重叠时间 = 冲突标记。
3. **容量预警**：某天总发送量超过渠道容量 → 预警。
**例子**：
```text
2026-06-01  ────────────────●───────────────────────────
            618大促预热      ●  会员日促销
                           ●
                           ●  冲突检测：同一用户群，
                           ●  建议调整时间或合并活动
```
***
## 缺失 5：死信队列（DLQ）与失败作业重放（运维能力）
### 问题
目前 Zeebe 有重试机制，但如果作业连续失败（如第三方 API 永久不可用），这些作业就会“卡住”，运维需要：
* 查看所有失败的作业
* 手动重放（Replay）
* 排查错误堆栈
### 应该放在哪里？
* 第11章（Execution Runtime）或第12章（生产架构）扩展
### 需要定义的内容
1. **失败作业列表**：提供管理界面，展示所有 `FAILED` 状态的 Job。
2. **错误分类**：
   * 可重试（临时网络问题）→ 自动重试
   * 不可重试（配置错误、业务逻辑错误）→ 人工介入
3. **手动重放**：选中失败的作业，重新触发执行。
4. **死信归档**：超过最大重试次数的作业归档到死信表，便于事后审计。
***
## 缺失 6：入站 Webhook（外部事件接收）
### 问题
目前事件驱动只监听 **Loyalty 内部事件**（Kafka）。但营销平台也需要接收**外部第三方事件**：
* CRM 系统推送“用户注册”
* 电商平台推送“退款申请”
* 线下门店推送“到店核销”
### 应该放在哪里？
* 第6章（Event System）或事件补全设计扩展
### 需要定义的内容
1. **外部事件接入点**：
   text
   复制
   下载
   ```
   POST /api/campaign/event/webhook/{programCode}/{eventType}
   ```
2. **鉴权机制**：API Key、签名验证（防伪造事件）。
3. **事件映射**：外部 JSON 格式 → 内部标准事件格式。
4. **可靠性保证**：接收后立即返回 200，异步处理（避免超时）。
***
## 缺失 7：多品牌/多 Program 的全局隔离与共享
### 问题
目前设计中 `program_code` 是隔离维度，但有些场景需要**跨 Program 共享**：
* 集团下有多个品牌，希望共享“黑名单”或“退订用户列表”。
* 跨品牌的联合营销活动（如：买A品牌送B品牌优惠券）。
### 应该放在哪里？
* 第12章（多租户架构）扩展
### 需要定义的内容
1. **全局黑名单**：跨 Program 共享的黑名单。
2. **跨 Program 活动**：一个 Campaign 可以关联多个 `program_code`。
3. **数据共享规则**：什么数据可以跨 Program 共享？需要什么权限？
***
## 缺失 8：动态内容与个性化推荐
### 问题
目前内容是“静态模板 + 变量替换”（如 `{{user_name}}`）。但现代营销需要**个性化推荐**：
* 给用户推荐“你可能喜欢的商品”
* 根据用户浏览历史动态生成内容
### 应该放在哪里？
* 第13章（Content）扩展
### 需要定义的内容
1. **推荐引擎集成**：调用推荐服务，获取该用户的 Top N 推荐商品。
2. **动态内容块**：邮件模板中的内容块由推荐引擎动态生成。
3. **个性化变量增强**：支持 `{{recommend.products}}` 这类动态变量。
***

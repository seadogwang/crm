### 1.2 核心理念：统一模型，多种触发
| 维度       | 批处理 Campaign | 事件驱动 Campaign      |
| -------- | ------------ | ------------------ |
| **触发方式** | 手动点击 / 定时任务  | Kafka 事件实时触发       |
| **用户来源** | 人群筛选（规则/分群）  | 事件携带（memberId）     |
| **执行频率** | 一次性 / 周期性    | 每次匹配事件触发           |
| **预算模式** | 总预算分配        | 单次成本 × 预估触发次数      |
| **生命周期** | 计划 → 执行 → 完成 | 部署 → 持续监听 → 停用     |
| **画布起点** | `START` 节点   | `EVENT_TRIGGER` 节点 |

### 3.3 实时机会处理器
```java
@Service
public class RealTimeOpportunityHandler {
    @Autowired
    private OpportunityRepository opportunityRepository;
    @Autowired
    private EventTriggerService eventTriggerService;
    /**
     * 事件到达时，判断是否形成“机会”
     * 如果事件满足条件，生成 Opportunity 记录并触发 Campaign
     */
    @Transactional
    public void handleEvent(JsonNode event) {
        String eventType = event.path("eventType").asText();
        String memberId = event.path("memberId").asText();
        // 1. 根据事件类型判断机会类型
        String opportunityType = mapEventToOpportunityType(eventType);
        // 2. 计算紧迫度
        double urgencyScore = calculateUrgency(event);
        // 3. 生成 Opportunity 记录
        Opportunity opp = Opportunity.builder()
                .memberId(memberId)
                .opportunityType(opportunityType)
                .source("REALTIME")
                .eventType(eventType)
                .eventPayload(event)
                .urgencyScore(urgencyScore)
                .score(0.8 + urgencyScore * 0.2)  // 综合评分
                .detectedAt(Instant.now())
                .status("ACTIVE")
                .build();
        opportunityRepository.save(opp);
        // 4. 触发匹配的 Campaign
        eventTriggerService.processEvent(eventType, memberId, event);
    }
    /**
     * 事件类型 → 机会类型映射（可配置）
     */
    private String mapEventToOpportunityType(String eventType) {
        switch (eventType) {
            case "CART_ABANDONED": return "CART_RECOVERY";
            case "ORDER_CREATED": return "CROSS_SELL";
            case "TIER_CHANGED": return "ENGAGEMENT";
            case "LOGIN_7_DAYS": return "LOYALTY_REWARD";
            default: return "EVENT_SIGNAL";
        }
    }
    /**
     * 计算紧迫度：越紧迫越应该立即触发
     */
    private double calculateUrgency(JsonNode event) {
        String eventType = event.path("eventType").asText();
        switch (eventType) {
            case "CART_ABANDONED": return 1.0;   // 最高紧迫
            case "ORDER_REFUNDED": return 0.9;   // 退款需要立即响应
            case "TIER_CHANGED": return 0.7;
            default: return 0.5;
        }
    }
}
```

### 4.2 冲突仲裁：事件驱动 vs 批处理
```java
@Service
public class UnifiedArbitrationEngine {
    /**
     * 统一仲裁：同一用户在同一时间段内，多个 Campaign 竞争
     */
    public Campaign resolveConflict(String memberId, List<Campaign> candidates) {
        // 1. 检查 Attention Budget（频控）
        for (Campaign c : candidates) {
            if (!attentionBudgetService.canSend(memberId, c.getChannel())) {
                continue;
            }
        }
        // 2. 按优先级排序
        candidates.sort((a, b) -> {
            // 事件驱动 Campaign 默认优先级更高（紧迫性）
            int priorityA = "EVENT".equals(a.getTriggerType()) ? 100 : 50;
            int priorityB = "EVENT".equals(b.getTriggerType()) ? 100 : 50;
            return Integer.compare(priorityB, priorityA);
        });
        // 3. 返回最高优先级
        return candidates.get(0);
    }
}
```

## 九、总结
事件驱动营销不是独立模块，而是**贯穿全平台的触发方式扩展**：
| 维度      | 融合方式                                                            |
| ------- | --------------------------------------------------------------- |
| **规划层** | 事件驱动 Campaign 与批处理 Campaign 共享 Workspace → Goal → Initiative 层级 |
| **机会层** | 事件作为“实时机会信号”，与批量机会共同存入统一机会池                                     |
| **决策层** | 预算分配支持“按次成本”模式，冲突仲裁统一处理两种触发方式                                   |
| **执行层** | Zeebe Message Start Event 实现事件触发，Workers 完全复用                   |
| **事件层** | 复用 Loyalty EventBridge，新增 Campaign Event Listener               |
| **前端**  | 统一活动列表，新建时选择触发方式                                                |
**核心理念**：事件驱动不是替代批处理，而是**补充**。两者在统一的平台框架下协同工作，覆盖用户全生命周期的营销需求。
